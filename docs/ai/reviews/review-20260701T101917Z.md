## Stage 2 — Code Quality Findings

### ERROR E1 — Reaper over-kill via path-prefix substring match

**File:** `.aai/scripts/aai-reap-tests.sh`
**Line:** 105-108 (workspace guard in the `while read` loop)
**Category:** Safety — violates the "never global" invariant from the spec

**Finding:**

The workspace guard uses a bare shell glob substring match:
```sh
case "$rest" in
  *"$WORKSPACE"*) ;;
  *) continue ;;
esac
```

If `WORKSPACE` is a path prefix of another workspace (e.g., `/home/user/aai` is a prefix of `/home/user/aai-v2`), any process whose command line contains `/home/user/aai-v2` also matches `*"/home/user/aai"*`, causing the reaper to kill that process even though it belongs to a completely different workspace.

**Breaking input (demonstrated, confirmed by live execution):**

```
WORKSPACE=/tmp/aai-review-test.sjacS6/myproject
Running process in /tmp/aai-review-test.sjacS6/myproject-fork with
  argv[0] = "vitest_/tmp/aai-review-test.sjacS6/myproject-fork_worker"

Result: AAI_REAP_WORKSPACE=/tmp/myproject → reaper output "reaped: 1"
→ The process in myproject-FORK was killed.
OVER-KILL CONFIRMED.
```

This violates the spec's SAFETY INVARIANT: "this reaper is NEVER global. It kills ONLY processes whose command line ... contains the current workspace path ... A matching process in a DIFFERENT workspace (a sibling checkout) is left alone."

The real danger: a developer with checkout `/home/user/aai` and `/home/user/aai-v2` running the loop on one would kill vitest processes in the other.

**Root cause:** Substring match does not enforce path-prefix semantics. The WORKSPACE path must be followed by a path separator (`/`) for the guard to be workspace-scoped.

**Suggested fix — in `aai-reap-tests.sh`:**

```sh
# Guard 2: must belong to THIS workspace — path-prefix anchored, never substring.
# Require the workspace path to be followed by '/' so that a checkout at
# /home/user/aai does NOT match a process in /home/user/aai-fork.
case "$rest" in
  *"${WORKSPACE}/"*) ;;
  *) continue ;;
esac
```

**Required companion fix — in `tests/skills/test-aai-run-tests.sh`:**

The spawn_marked helper currently creates markers like `vitest_run_${ws}_worker` where `ws` is a directory path followed immediately by `_worker`. With the corrected guard (`*"${ws}/"*`), `_worker` does not match `/`. Update the markers to embed the workspace as a proper path component:

```bash
spawn_marked() {
  local argv0="$1"
  local ws="$2"  # workspace path
  # Embed workspace as a real path so *"$WORKSPACE/"* matches:
  bash -c 'exec -a "$0" sleep 600' "${argv0}/${ws}/config_worker" >/dev/null 2>&1 &
  ...
}
```

Or, simpler: append a trailing `/` to the workspace path when calling spawn_marked:
```bash
match_pid="$(spawn_marked "vitest_run_${ws}/worker")"
other_pid="$(spawn_marked "vitest_run_${other}/worker")"
```

And set `AAI_REAP_WORKSPACE="${ws}"` (without the trailing slash; the guard code adds `/` implicitly via `*"${WORKSPACE}/"*`).

**NOTE:** The fix must also ensure that the `WORKSPACE` variable itself never ends in a trailing slash before the case comparison (otherwise `*"${WORKSPACE}/"*` becomes `*"/path/to/ws//"*` which would not match `"/path/to/ws/config"`). A safe approach:

```sh
WORKSPACE="${AAI_REAP_WORKSPACE:-$PWD}"
WORKSPACE="${WORKSPACE%/}"    # strip any trailing slash once
# Then use *"${WORKSPACE}/"* in the case guard
```

### INFO I1 — Watchdog TIMED_OUT_FILE race (very narrow)

**File:** `.aai/scripts/aai-run-tests.sh`
**Lines:** 56-84

The script creates a temp file (`mktemp`), then immediately removes it, then uses it as a signal marker that the watchdog touches if it fires. If the command exits naturally at the exact moment the watchdog fires (e.g., the command runs for exactly `TIMEOUT` seconds), the watchdog might write the file AND the main `wait` returns — and the main script then reads `TIMED_OUT=1` and exits 124 even though the command completed normally. This produces a spurious exit 124 in an extremely narrow race window (sub-millisecond). The correct exit code is lost.

No fix needed unless deterministic timeout-vs-natural-exit discrimination is required. INFO only.

---

### INFO I2 — 1-second overhead on every wrapper invocation (even fast successes)

**File:** `.aai/scripts/aai-run-tests.sh`
**Lines:** 90-92

The group-kill cleanup block always runs `sleep 1` between TERM and KILL regardless of whether the command succeeded (no survivors) or timed out. Every invocation of the wrapper takes at minimum 1 extra second. For fast unit tests (< 1s per run), the overhead is proportionally large.

This is by-design (grace period for TERM before KILL) but adds 1 second to every invocation. No safety concern; the spec (D2) requires TERM-then-KILL. INFO only.

---

### INFO I3 — No path-metacharacter guard for WORKSPACE in `case` pattern

**File:** `.aai/scripts/aai-reap-tests.sh`
**Lines:** 105-108

If `$WORKSPACE` contains shell glob metacharacters (`*`, `?`, `[`, `]`), they are treated as pattern characters inside the `case "$rest" in *"$WORKSPACE"* )` expression. For example, `WORKSPACE="/home/user/proj[ect]"` would create a character class `[ect]` matching one character `e`, `c`, or `t`, breaking the match semantics.

In practice, workspace paths are filesystem paths that rarely contain these characters. No fix needed unless hardening against adversarial input is required.

---

### INFO I4 — bootstrap `add_unique` bash-3.2 empty-array fix is functionally correct but slightly fragile

**File:** `.aai/scripts/aai-bootstrap.sh`
**Lines:** 81-97

The fix to handle bash-3.2 empty arrays (using `eval` + length check before expansion, then `"${current[@]:-}"`) is functionally correct: `local current=()` explicitly declares the array, so `set -u` does not fire; and `"${current[@]:-}"` expands to an empty word when `current` is empty, which the `for` loop iterates once with `existing=""`, correctly comparing `"" == "$value"` as false. Tests pass (including the bootstrap test suite). 

The use of `eval` to work around bash array-by-reference limitations is correct but verbose. Future bash refactors could use bash-4.3 `declare -n` nameref instead. INFO only.

## Adversarial Focus Summary (per review brief)

### Focus #1 — Reaper over-kill (DANGEROUS): FAIL

**Can `aai-reap-tests.sh` EVER kill a process outside the current workspace?**

YES — demonstrated. WORKSPACE as a string prefix of another workspace's path causes over-kill. The current `*"$WORKSPACE"*` glob matches any string containing the WORKSPACE path, including paths where WORKSPACE is a directory-name prefix. See ERROR E1 above.

Test-005 does NOT catch this because mktemp generates non-overlapping temp directory names (they share a common parent but the names themselves are different random suffixes like `ws.ABC` vs `other.DEF`).

### Focus #2 — Wrapper group-kill correctness: PASS

**Is the pgid always the child's own new group?**

YES. With `set -m` (job control enabled), `"$@" &` creates the child in a new process group with pgid == child's pid. The `PGID="$CMD_PID"` is always the child's group, never the calling shell's group. `kill -TERM -"$PGID"` cannot kill the calling shell.

**Can the watchdog kill an unrelated process via a reused pgid?**

Extremely unlikely. The pgid is set before the watchdog starts and cleared after the watchdog exits. The window is short and pid reuse within it is theoretical only.

**Is there TERM→KILL escalation?**

YES. Lines 90-92: `kill -TERM -"$PGID"`, `sleep 1`, `kill -KILL -"$PGID"`. Both wrapper and reaper implement the escalation.

**Does the timeout watchdog get reaped?**

YES. Lines 78-79: `kill "$WATCHDOG_PID"; wait "$WATCHDOG_PID"`. The watchdog is always stopped and waited before the cleanup kills run. No late-firing watchdog risk.

**Exit code fidelity?**

CORRECT. Success → 0 (real status), failure → N (real status), timeout → 124 (via TIMED_OUT file check). The spec notes exit 124 is the GNU-timeout convention; callers should note that a command exiting 124 coincidentally is indistinguishable from a true timeout (same limitation as GNU timeout; not a defect).

**Are real exit codes preserved (128+signal)?**

YES. `wait "$CMD_PID"` returns 128+signal when the command is killed by a signal. STATUS captures this. If the watchdog fired (TIMED_OUT=1), we exit 124 instead, discarding the 128+signal status. This is correct per the spec (timeout = 124, not "killed by signal N"). The TERM the watchdog sends produces STATUS=143 (128+15), which is then overridden to 124. This is correct.

### Focus #3 — Quoting / word-splitting / injection: PASS

- `"$@"` passthrough in the wrapper is correct. No word-splitting or injection risk.
- `$WORKSPACE` is double-quoted in the `case` pattern: `*"$WORKSPACE"*`. Word-splitting is not triggered in `case` patterns. Spaces in WORKSPACE are handled correctly (the double-quotes make them literal).
- `$PWD` is not used directly in `pkill` or any other expansion beyond the `case` guard.
- `while read -r pid etime rest` correctly handles spaces in the `rest` (args) field since `read` with three variables places all remaining words into `rest`.

### Focus #4 — Portability (macOS bash 3.2 + BSD ps): PASS (with note)

- The wrapper uses `#!/bin/sh` with `set -m`, `kill`, `mktemp`, `sleep` — all POSIX.
- No GNU-only `timeout` command; the inline watchdog replaces it. Correct.
- `ps axo pid=,etime=,args=` is BSD-compatible. On macOS when output is a pipe/file (not a terminal), command lines are NOT truncated (confirmed by live test with 100+ char argv[0] showing full line). The truncation concern raised in the spec's edge-case section does not manifest in practice on this macOS version.
- `kill -TERM -"$PGID"` and `kill -KILL` are POSIX-standard.
- The reaper's `etime_to_secs` function handles all four ps etime formats: SS, MM:SS, HH:MM:SS, D-HH:MM:SS. The `10#0${...}` prefix correctly prevents octal interpretation of leading zeros. BSD and GNU ps both use these formats.

### Focus #5 — Bootstrap bash-3.2 empty-array fix: PASS

The `add_unique` function correctly handles bash 3.2 empty arrays (see INFO I4). The fix is functionally sound. The bootstrap test suite passes.

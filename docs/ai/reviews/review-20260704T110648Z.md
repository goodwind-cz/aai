### WARNING-3 [LINT-MASKING-EDGES] .aai/scripts/lib/docs-audit-core.mjs:302-321 (maskInlineCode) and :341-357 (fence open) — CONFIRMED (probes), no corpus impact
**Category:** Correctness / false-positive posture (D1: "NEVER flags content inside either")
Two CommonMark divergences found by construction, both violating D1's conservative
contract in corner shapes:
(a) Multi-line inline code span: ``prose `code / </content> / more` end`` is ONE code
span in CommonMark; per-line masking cannot see it, so the middle line is flagged
`stray-tool-markup` — a false positive at intake `--strict` (hard-fails the save).
(b) A line-initial two-run code span (``` ``` x ``` ``` on one line) is a paragraph +
code span in CommonMark (backtick-fence info strings may not contain backticks), but the
fence tracker reads it as a fence OPEN: probe yields a spurious `unbalanced-fence` AND
swallows every subsequent line (any later real residue is hidden — a false-negative
cascade). The real corpus is clean under both (TEST-009 green; `--lint-body` = 0), so
this is latent, not active. Cheap hardening: treat a line whose post-run remainder
contains a backtick as non-fence (CommonMark rule), and accept that multi-line spans
stay out of the masking model (document it in the D1 comment).
**Disposition (H6):** promote to follow-up ref — proposed intake "lintBody CommonMark
edge hardening (multi-line spans, backtick-in-info-string)".

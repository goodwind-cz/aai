## Detailed Findings

### INFO I-A — Backslash paths: POSIX-correct, theoretical Windows risk (not a safety error)

**Input:** `"apps\\api"` (backslash, single segment after `split('/')`) vs `"apps/api"`.
**Selector output:** `mode=parallel`, both co-scheduled.
**Why this is NOT a safety error on this repo:** On POSIX (macOS/Linux), `\` is a valid filename character, not a path separator. The path `apps\api` refers to a file literally named `apps\api` in the current directory — a different filesystem object from `apps/api/`. The selector correctly treats them as non-overlapping because they genuinely are different paths. Since this repo runs on POSIX (darwin, confirmed), backslash paths are not in scope.
**Theoretical risk:** An orchestrator on Windows — or copy-pasted Windows-style paths — would silently miss the overlap. The spec and selector operate exclusively in the POSIX domain (forward-slash paths). No code change needed; noting for documentation only.
**Classification:** INFO (not ERROR, not WARNING). POSIX-correct behavior; out of scope for this repo.

### INFO I-B — Whitespace/tab-only path segments: POSIX-correct, different real paths

**Input:** `"apps/ /api"` or `"apps/\t/api"` vs `"apps/api"`.
**Selector output:** `mode=parallel`, both co-scheduled.
**Why this is NOT a safety error:** On any POSIX filesystem, a directory segment of `" "` (space) or `"\t"` (tab) is a valid, distinct directory name different from the absence of that segment. `apps/ /api` and `apps/api` are genuinely different paths; there is no overlap to detect. If an operator accidentally types `"apps/ /api"` meaning `"apps/api"`, that is an operator error in the source spec — the paths are still genuinely different on disk.
**Note:** The trim() call only strips leading/trailing outer whitespace on the full string, not inner segment whitespace. This is correct behavior.
**Classification:** INFO. No code change needed.

### INFO I-C — Duplicate-id and self-parent edge cases lack a dedicated test (carried from prior review I1)

`augUncertain` / `describeUncertain` (lines 159-167, 189-192) correctly mark duplicate-id scopes and self-parent scopes as uncertain and fail-closed. The logic is correct and unchanged. No dedicated TEST-xxx exercises this code path; the prior review noted it as I1. Still no dedicated test. Throughput-only risk if these edge cases are ever exercised.

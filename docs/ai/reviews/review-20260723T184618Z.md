## Code quality findings

### BLOCKING — supersession wrongly suppresses genuine false-opens whose only delivery evidence is untimed (Arm A / D2c)

`.aai/scripts/lib/docs-audit-core.mjs:361-400`. The supersession block (lines
excerpted):

```js
if (evidenced) {
  const lifecycle = events.filter(e => e.event === 'doc_lifecycle'
    && e.ts && idCandidates.some(c => idRef(e.ref, c)));
  if (lifecycle.length) {
    const latest = lifecycle.reduce((a, b) => (String(b.ts) > String(a.ts) ? b : a));
    const to = String(latest.payload?.to ?? '').toLowerCase();
    if (FALSE_OPEN_STATUSES.has(to)) {
      let deliveryTs = events
        .filter(e => (e.event === 'work_item_closed' || e.event === 'ac_evidence')
          && e.ts && idCandidates.some(c => idRef(e.ref, c)))
        .reduce((max, e) => (String(e.ts) > max ? String(e.ts) : max), '');
      for (const c of idCandidates) {
        const fts = flushDateToTs(metricsFlushes.get(c));
        if (fts > deliveryTs) deliveryTs = fts;
      }
      if (!deliveryTs || String(latest.ts) > deliveryTs) {
        return { evidenced: false, reasons: [...] };
      }
    }
  }
}
```

`deliveryTs` is built from exactly two sources: `work_item_closed`/`ac_evidence`
event timestamps, and a METRICS flush `date_utc`. It never incorporates a
timestamp for the delivery-commit arm (Arm A, the ORIGINAL SPEC-0039/CHANGE-0027
signal — a plain `feat:` commit mentioning the doc id, with no close ceremony
ever run) or the AC-status-table arm (D2c). Neither of those two signals has a
timestamp to contribute in this codebase's ledger. When a doc is `evidenced`
ONLY through one of those two arms, `deliveryTs` stays `''`, and
`!deliveryTs` at line 397 unconditionally supersedes — the doc is de-flagged
as soon as ANY `doc_lifecycle` event ever moved it to an open status, with
no chronological check against the real delivery at all.

**Live repro** (isolated fixture, `setup_iso_repo`-equivalent, run today):

```
$ cat docs/issues/CHANGE-7001-repro.md   # status: draft
$ node .aai/scripts/append-event.mjs --event doc_lifecycle \
    --ref CHANGE-7001 --from done --to draft      # reopen FIRST
$ echo shipped > DELIVERY.md && git commit -m "feat: deliver CHANGE-7001 to production"
                                                    # real delivery commit SECOND (Arm A)
$ node .aai/scripts/docs-audit.mjs --no-event
...
- Scanned: 1 docs | ... | False-open: 0 ...
### Drift report: 0
_None._
```

CHANGE-7001 is a textbook false-open (delivered via commit, frontmatter never
flipped, reopen predates delivery — the exact scenario the spec's AC-02 text
says must STILL flag) and is silently unflagged. This is not a synthetic
edge case: `.aai/SKILL_LOOP.prompt.md:32` documents `doc_lifecycle` as
emitted "whenever a doc's frontmatter `status` field changes" (not only at
close), and `.aai/VALIDATION.prompt.md:155` appends one on every non-`done`
status change during ordinary validation/remediation. Any doc that cycles
through an open status during normal loop operation and is later delivered
by a plain commit (no `work_item_closed`/`ac_evidence`/METRICS event) loses
false-open coverage permanently, independent of when the reopen actually
happened relative to delivery — which is the precise "over-eager
supersession blinds the audit to genuine false-opens" failure class the
spec's own Design decisions D2 and the dispatch instructions call out as
highest-severity.

**Why the test suite didn't catch it**: `test_fometrics_supersession`
(TEST-002) sub-cases e/f/g/h all route delivery evidence through
`work_item_closed` or a dated METRICS flush — every case that *is* covered
by `deliveryTs`. No sub-case combines Arm A (delivery-commit) or D2c
(AC-status-table) evidence with a `doc_lifecycle` event, so the gap shipped
untested; the SPEC-0039 positive control `CHANGE-9001` (Arm-A-only,
`setup_fo_repo`) is reused by `assert_fo_control_flagged` in both new test
functions but never combined with a `doc_lifecycle` event either.

**Fix shape** (not applied — read-only review): either (a) extend
`deliveryTs`'s event filter to also cover the `docs_audit`/commit signal via
some ts proxy, which doesn't exist for git commits in this codebase, so the
more consistent fix is (b) flip the fail-closed default — when `deliveryTs`
cannot be established at all (`!deliveryTs`), do NOT supersede (leave the
existing `evidenced` result untouched) rather than always superseding;
supersession should require a *provably* newer reopen, never merely an
unprovable one. That directly matches the dispatch's stated correctness
bar: "supersede ONLY when... its ts is strictly greater than deliveryTs" —
an unknown deliveryTs can never be proven smaller than the reopen's ts, so
the honest answer under the stated rule is "do not supersede."

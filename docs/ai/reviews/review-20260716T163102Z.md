## Verdict 2 — code_quality: FAIL

One BLOCKING finding (F1) — data-loss class in the distribution channel:

- F1 (BLOCKING) aai-sync.sh:292 — unquoted prefix-strip pattern. In bash,
  `${tgt#$DST_ROOT/}` treats the expansion of `$DST_ROOT` as a glob PATTERN.
  A target path containing `[`, `]`, `*`, or `?` (e.g. `~/projects/tgt[1]`)
  breaks the strip; `rel` remains absolute; every guard keyed on `rel`
  misses; the prune deletes the entire just-copied `.aai` tree, the user's
  target-only scripts, and `.aai/cache` — then exits 0. Reproduced live.
  The ps1 twin is immune (literal `Substring`, aai-sync.ps1:262), so this is
  also a sh/ps1 behavior divergence. One-line fix plus a special-char-path
  regression test (RED-provable against the current code).
- F2 (NON-BLOCKING) aai-sync.sh:73 vs aai-sync.ps1:121 — sticky-pin
  trailing-whitespace tolerance differs; sh silently downgrades a core pin to
  extended (full-layer reinstall), ps1 honors it. Reproduced.
- F3 (NON-BLOCKING) aai-sync.sh:104 (and the same awk in
  tests/skills/test-aai-layer-profiles.sh:61) vs aai-sync.ps1:143 — manifest
  entry trailing-whitespace handling differs; sh silently ships a core
  install minus the affected file, ps1 ships it. Reproduced; canonical-repo
  exposure is gated by TEST-001 (same untrimmed awk → conformance fails
  loudly), so residual risk is downstream-edited manifests.

INFO (non-gating, no failure mode worth a finding):
- aai-sync.sh:309 / ps1 empty-dir cleanup can remove an EMPTY `.aai/cache`
  directory despite the "never cache/" comment — files in cache are safe
  (prune guard) so the dir only disappears when it holds nothing; consumers

- ps1 parameter `$Profile` shadows PowerShell's automatic `$PROFILE`

- A CRLF-converted PROFILES.yaml makes the sh awk find no `core:` section —

  F2/F3.

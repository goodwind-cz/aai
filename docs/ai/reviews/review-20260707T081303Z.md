## Stage 2 — Findings

### ERROR E1 — `--clear` whitelist is open to JS prototype-chain field names: unknown fields exit 0 and WRITE junk into STATE
**Category:** Security / input validation (write-path integrity of the loop's own state writer)
**File:** `.aai/scripts/state.mjs:516-517` (`const info = spec[f]; if (!info) …`), reached from all four `--clear` subcommands; `applyClears` at `:535-541` then performs the write.

`resolveClearList` tests membership with plain property access on an object literal, so every inherited
`Object.prototype` member name passes the "closed" whitelist:

```
$ node .aai/scripts/state.mjs set-worktree --clear toString --state fx.yaml   # exit 0
worktree:
  ...
  toString: null        <- junk field written into the live-schema block
$ node .aai/scripts/state.mjs set-worktree --clear __proto__ --state fx.yaml  # exit 0 -> "__proto__: null"
$ node .aai/scripts/state.mjs set-validation --clear valueOf ...              # exit 0 (all four subcommands affected)
```

`check-state.mjs` accepts the polluted file (exit 0), so the junk key persists in STATE across the loop.
This directly violates:
- Spec-AC-03 ("unknown field → exit 2 … no write on rejection") — `toString` is not in any documented clearable set;
- the engine's own exit-code contract (state.mjs:48-51: "usage/validation error BEFORE ANY WRITE (… unknown …)");
- the closed-set discipline D2 explicitly builds on.

Bounded blast radius (verified): no verdict/status/policy field is reachable — `status`, `required`,
`recommendation`, `user_decision`, `run_at_utc` are not `Object.prototype` members and stay refused (probe:
`set-validation --clear status` → exit 2 naming reset-block). The reachable damage is arbitrary
prototype-named junk keys written into `worktree` / `code_review` / `last_validation` / `current_focus`.
Decision logic is not corruptible through this hole, but the strict-validation invariant IS: an input the spec
mandates be refused pre-write instead mutates STATE with exit 0.

**Fix (small):** own-property check — `const info = Object.hasOwn(spec, f) ? spec[f] : undefined;`
(or `Object.prototype.hasOwnProperty.call(spec, f)`; or build `CLEAR_FIELDS` maps with
`Object.create(null)`). Add a regression case to test_029 asserting `--clear toString` and `--clear __proto__`
exit 2 byte-identical.

### WARNING W1 — repeated `--clear` flags silently drop the earlier list (last-wins)
**Category:** Best practices (the exact "silent drop" class the W5 strict-flag hardening targets)
**File:** `.aai/scripts/state.mjs:117` (`MULTI_FLAGS` lacks `clear`) + `:127-132` (last-wins), consumed at `:502`.

Probe: `set-worktree --clear branch --clear path` → exit 0, `path: null`, but `branch: feat/stale` SURVIVES.
The operator's intent ("clear both") half-executes silently — re-introducing the very stale-field leak F1
exists to eliminate, with a success exit code. Last-wins is consistent with other scalar flags, but for other
flags the loser is a redundant VALUE; here the loser is a whole dropped INSTRUCTION.
**Fix:** add `clear` to `MULTI_FLAGS` and merge the comma-lists in `resolveClearList`
(`[].concat(raw)`…), or refuse a repeated `--clear` with exit 2.

### WARNING W2 — clearing a hand-edited `>-` field containing a blank line orphans continuation lines and yields a junk non-null value
**Category:** Best practices / data integrity (W1-of-SPEC-0012 class; pre-existing helper, newly reachable surface)
**File:** `.aai/scripts/state.mjs:349-358` (`fieldSpan` stops at the first blank line), reached via `applyClears` → `setField` (`:414-427`).

Probe: `code_review.notes: >-` hand-edited to hold two blank-line-separated paragraphs; `set-code-review
--clear notes` → `notes: null` followed by a blank and the orphaned `    para two line`. PyYAML parses the
result as `notes: 'null\npara two line'` — the clear produced a junk STRING, not null, and `check-state`
exit 0 hides it. In-contract inputs are safe (this engine's `textFieldLines` never writes blank lines inside
`>-`), but STATE is hand-editable and F1's whole purpose is cleaning hand-observed stale fields.
Same hazard pre-exists in every `setField` overwrite (e.g. `--notes`), so this is a helper-level class, not a
`--clear`-specific regression.
**Fix option:** make `fieldSpan` continue across blank lines while the following line is more-indented than
the field (block-scalar span), or refuse (exit 1) to rewrite a field whose span is followed by a
deeper-indented continuation after a blank.

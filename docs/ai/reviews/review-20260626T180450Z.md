## STAGE 2 — Code Quality

### ERROR

#### E1 — Expired-reclaim TOCTOU: two acquirers can BOTH win one scope
**File:** `.aai/scripts/docs-lock.mjs:125-149` (the EEXIST/expired self-heal branch, esp. lines 132-137)

The acquire retry loop self-heals an expired lock by **blindly unlinking the lock
path** and retrying the O_EXCL create:

```js
const existing = readLock(p);
if (isExpired(existing)) {
  try { fs.rmSync(p, { force: true }); } catch { /* already gone */ }
  continue;                      // retry openSync(p, 'wx')
}
```

`fs.rmSync(p, {force:true})` removes whatever currently sits at `p` — it is **not**
conditioned on the lock still being the expired one this acquirer observed. That
opens a reachable double-claim window:

```
1. A: openSync('wx') -> EEXIST; readLock -> L0 (expired)
2. B: openSync('wx') -> EEXIST; readLock -> L0 (expired)
3. A: rmSync(L0); retry openSync('wx') -> SUCCESS; writeSync L_A; return EXIT 0   (winner #1)
4. B: rmSync(p) -> UNLINKS A's FRESH L_A (by path, unconditionally)
5. B: retry openSync('wx') -> SUCCESS (file gone); writeSync L_B; return EXIT 0   (winner #2)
```

Both A and B print `acquired` and exit 0 for the same scope; only L_B survives and
A's lock is silently destroyed — exactly the lost-update / two-orchestrators-on-one-scope
failure RFC-0004 exists to prevent. This violates Spec-AC-03 and contradicts D4's
"the atomic create is the sole arbiter … double-claim remains impossible." D4's
reasoning only covers "both delete, then both race the create"; it misses the
interleaving where one acquirer **completes its create before the other's blind
unlink fires**.

**Evidence (empirical, this review):**
- Blind N=40 / 12-iteration stress pre-seeded with an expired lock did NOT trip it
  (window is a couple of syscalls wide under real process jitter) — rarity, not safety.
- A copy of the script (product code untouched) with a delay injected at the exact
  vulnerable point (after `isExpired()`, before `rmSync`) — simulating a legal OS
  scheduler preemption — produced **two exit-0 winners**, with owner-A's lock
  destroyed and final owner = owner-B. This proves the interleaving is logically
  reachable; the product code differs only in the probability of that scheduling.

**Recommendation — make the reclaim itself single-winner (keep O_EXCL as create-arbiter):**
Replace the blind `rmSync + continue` with an **atomic rename-steal**. Each acquirer
attempts to move the stale lock aside under a name unique to itself:

```js
const aside = `${p}.stale-${process.pid}-${Math.random().toString(36).slice(2)}`;
try {
  fs.renameSync(p, aside);     // atomic; removes source name
} catch (e) {
  if (e.code === 'ENOENT') continue; // someone else already stole/replaced it -> re-loop
  throw e;
}
fs.rmSync(aside, { force: true });   // we exclusively own the stale file now
continue;                            // retry openSync(p,'wx'); we created the gap
```

`rename(2)` is atomic and removes the source name, so for a given inode only ONE
concurrent acquirer can succeed; the losers get `ENOENT` and re-loop, where they
either find a fresh lock (a rename-winner's new lock) and fail closed exit 3, or
find the path empty and race the O_EXCL create normally. This closes the window
without weakening the free-scope path. (Equivalent alternative: gate reclaim behind
an OS advisory lock on a sidecar file, but the rename-steal stays within the
node:fs-only constraint of D5.) Whichever fix lands, add the W1 concurrency test.

### WARNING

#### W1 — TEST-003 never exercises the expired-reclaim concurrency path
**File:** `tests/skills/test-aai-docs-lock.sh:137-176`

TEST-003 calls `fresh_lockdir` (empty dir) before launching N acquires, so every
acquirer hits the O_EXCL create directly and the `rmSync` self-heal branch is never
entered. The validator's 10-iteration stress (per STATE notes) likewise started from
free scopes. That is precisely why E1 passed TDD and validation green. Add a
concurrency test that **pre-seeds an already-expired lock**, launches N acquires of
that scope, and asserts exactly one exit 0 and exactly one lock file. (This new test
should be RED against the current blind-rmSync code under an injected preemption and
GREEN after the E1 fix.)

#### W2 — `safeScope` is many-to-one; distinct scopes can collide to one lock file
**File:** `.aai/scripts/docs-lock.mjs:51-57`

`safeScope` maps every `[^A-Za-z0-9._-]` char to `_`, so e.g. `a/b`, `a:b`, and
`a_b` all resolve to `a_b.lock`. Two genuinely-distinct scope ids that differ only
in sanitized characters would share one lock — manifesting as false contention
(exit 3) or a cross-scope wrong-owner (exit 4) for an unrelated scope. Unlikely for
disciplined AAI ref-id scopes (SPEC-0004, page names) but unbounded for free-form
scope strings. Consider appending a short hash of the original scope to the filename,
or detecting a collision (stored `scope` != requested `scope`) and failing 2.

> Path-traversal note (NOT a finding): `safeScope` neutralizes `/` (and all other
> separators) to `_`, and surviving `.`/`..` tokens cannot escape `LOCKS_DIR` via
> `path.join` (`..` -> filename `...lock`). A scope like `../../etc/passwd` becomes
> `.._.._etc_passwd.lock` inside the lock dir. No traversal risk.

## Expired-reclaim TOCTOU — explicit assessment

**NOT SAFE.** `fs.openSync(p,'wx')` is the sole arbiter only for the *create* step.
The self-heal *reclaim* step (`rmSync(p,{force:true})`) is an unconditional
unlink-by-path that is NOT serialized, so a slow acquirer can delete a fresh lock a
fast acquirer just created and then create its own — yielding two exit-0 winners.
Demonstrated empirically by injecting a legal scheduler preemption at the vulnerable
point (two winners, owner-A clobbered). The fix is to make the reclaim single-winner
via an atomic rename-steal (above), after which O_EXCL remains the create-arbiter and
the reclaim cannot clobber a fresh lock.

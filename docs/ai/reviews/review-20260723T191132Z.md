## Delta walk-through

### `commitDateTs()` (docs-audit-core.mjs:242-251)

```js
function commitDateTs(root, hash) {
  const iso = git(root, `show -s --format=%cI ${hash}`);
  if (!iso) return '';
  const d = new Date(iso.trim());
  return Number.isNaN(d.getTime()) ? '' : d.toISOString();
}
```

- Routed through the pre-existing `git()` helper (line 157), which wraps
  `execSync` in try/catch and returns `null` on any failure (bad hash,
  detached/missing object, non-repo, git binary error) â€” `stdio:
  ['ignore','pipe','ignore']` also suppresses stderr, so a git failure can
  never throw or leak noise into the audit's stdout. `commitDateTs` treats
  that `null` (falsy) as `''`, i.e. "no contribution" â€” fail-closed as
  documented.
- `%cI` is git's strict-ISO committer-date format (always offset-bearing,
  e.g. `2026-07-23T14:30:00+02:00`, never a bare non-parseable string for a
  valid commit); `new Date(iso.trim())` parses it correctly regardless of
  the source offset, and `.toISOString()` re-normalizes to UTC `Z`. The
  `Number.isNaN(d.getTime())` guard covers a theoretical unparseable output
  without throwing.
- Runs exactly once per unique hash: `deliveryHashes` is a `Set` (deduped
  across both `id`- and `fileId`-sourced commit lists, same as round 1's
  `hashes` local, just hoisted to function scope), and the fold loop
  (`for (const hash of deliveryHashes) { const cts = commitDateTs(...); ... }`)
  iterates it once. No redundant subprocess spawns for a repeated hash. It
  only runs at all inside the already-narrow guard (`evidenced` true, a
  `doc_lifecycle` event exists, latest `to` is an open status) â€” not on
  every doc in the corpus.
- Shell-injection surface: `hash` is sourced exclusively from git's own
  `%H` output in `deliveryCommitsForId` (40 lowercase hex chars) â€” never
  attacker- or filesystem-controlled content â€” same trust boundary already
  accepted elsewhere in this file (e.g. `cellHasDeliveryCitation`'s
  `rev-parse --verify` probe).

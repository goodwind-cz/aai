## Code quality — findings

### B1 (BLOCKING) — rollback bypassed when INDEX regeneration fails post-apply
`.aai/scripts/close-work-item.mjs:315` (via `regenerateIndex` :278-287, called
from the post-apply `selfVerify` at :436 inside the try block whose catch at
:444 owns rollback). `process.exit(1)` is not catchable, so a regen failure
AFTER apply skips `rollback()` entirely and leaves the mutated doc frontmatter
+ appended EVENTS lines on disk with exit 1. Directly violates D6.5 / Spec-AC-04
("No half-closed doc is ever left on disk"). Reachable via the spec's own R4
(downstream repo missing `generate-docs-index.mjs`) or any pre-existing
malformed doc that makes index generation throw. No test covers this sub-path
(TEST-008 exercises the audit-NOT-CLEAN branch, where rollback does run).
Recommended disposition: **remediate-in-tree** — make `regenerateIndex` signal
failure to the caller instead of exiting (throw, or return a problem string that
`selfVerify` surfaces), so the post-apply path rolls back before exiting; the
pre-write / idempotency-short-circuit call sites (:403) can keep exiting since
nothing was written there. Add a regression test (rig a missing/throwing
generate-docs-index in the fixture, assert exit != 0 AND doc + EVENTS
byte-identical).

### B2 (NON-BLOCKING) — malformed mixed inline+block YAML for non-empty inline links
`.aai/scripts/close-work-item.mjs:237`. See failure_scenario above. Recommended
disposition: **remediate-in-tree** (convert an inline non-empty list to block
form before appending) or **promote-to-follow-up ref** if judged too rare to fix
now. Low probability because the script itself only ever writes block form.

### B3 (NON-BLOCKING) — null fm.id resolved by display-id yields internal error, not a clean usage error
`.aai/scripts/close-work-item.mjs:344`. Fail-closed (bytes restored) but poor
diagnostics + a wasted apply/rollback. Recommended disposition:
**remediate-in-tree** (reject a resolved doc whose `fmId` is null with exit 2
before any write) or **promote-to-follow-up ref**. Low probability (repo
convention guarantees fm.id).

code_quality = **fail** (B1 BLOCKING).

## Transaction-correctness notes (dispatch specific-attention items)
1. EVENTS.jsonl rollback via `ftruncate` to the snapshot byte-length is
   byte-safe: `append-event.mjs` only ever `appendFileSync`s complete
   `JSON.stringify(entry)+'\n'` lines, the snapshot length is captured at the
   right moment (:415, after plan, before apply), and no other writer runs. The
   doc-file rollback restores full original bytes. Byte-perfect — EXCEPT it can
   be skipped entirely via B1.
2. Self-verify oracle runs the REAL `runAudit` over the mutated tree and any
   non-`tracked-done`/non-`aligned` classification, a not-found ref, OR a
   `missing-close-telemetry` hit reliably triggers rollback — for the covered
   path. Scoped to the closed refs only, which is spec-intended.
3. Idempotency: a partially-closed prior state (status done but a missing event)
   is REPAIRED, not silently accepted — `needsClosedEvent`/`needsAcEvidence`
   flip `anyMutation` true and the missing event is emitted.
4. Ref-form is correct for all doc types because the script always uses whatever
   `fm.id` is and the audit computes identity as the same `fm.id ?? ids.primary`
   — the two sides can't diverge (self-consistent even for a numbered fm.id).
5. Wiring placement and merge boundary are correct; no residual contradictory
   hand-close prose.
6. Fail modes: unresolvable/ambiguous ref, `--commit` missing, non-done-terminal
   status all exit non-zero pre-write. EVENTS-unwritable would surface via the
   emit/execFileSync catch -> rollback (except the B1 regen-failure hole).

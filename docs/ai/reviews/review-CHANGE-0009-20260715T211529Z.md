**W1 (fix-before-merge; confirms validation Finding 1) — vacuous self-containment guard, and the naive fix self-fails.**
`tests/skills/test-aai-docs-audit.sh:3366-3369`. `setup_fixture()` does a bare `cd "$TEST_DIR"` (line 83,
pre-existing), so when invoked as `bash tests/skills/test-aai-docs-audit.sh` the relative
`"${BASH_SOURCE[0]}"` no longer resolves by the time the guard runs: `grep` exits 2 (unreadable file), the
`&& log_fail` never fires (errexit exempts non-final commands in a `&&` list), and both guard lines pass
vacuously regardless of content. Worse, if the path were fixed, the second guard
(`grep -qF "SPEC-DRAFT-slug-refs""-across-tooling"`) would MATCH the new explanatory comment at line
~3355-3356, which spells out the full deleted path — the suite would fail on its own documentation.
**Exact fix:** (a) capture an absolute self-path at the top of the script before any `cd`
(`SUITE_FILE="$SCRIPT_DIR/$(basename "${BASH_SOURCE[0]}")"` — `SCRIPT_DIR` is already computed absolute)
and guard readability first (`[[ -r "$SUITE_FILE" ]] || log_fail "self-containment guard cannot read
$SUITE_FILE"`); (b) use `"$SUITE_FILE"` in both greps; (c) de-literalize the comment (paraphrase the
deleted filename, e.g. "the original hardcoded slug-refs DRAFT path") OR strip comments before matching
(`sed 's/[[:space:]]*#.*$//' "$SUITE_FILE" | grep -qF "SPEC-DRAFT-slug-refs""-across-tooling"`).
Does not change Spec-AC-10's outcome (core fixture fix independently sound), but TEST-019's "no hardcoded
repo DRAFT path" claim is not machine-proven until fixed.

**W2 — guard-config conformance test misses real, already-present grep/reader divergences.**
`lib/guard-config.mjs:64` requires column-0 keys; both hook greps
(`.aai/scripts/pre-commit-checks.sh:215`, `.aai/scripts/install-pre-commit-hook.ps1:115`) accept
`^[[:space:]]*` leading whitespace. Divergences that exist TODAY: (a) an indented
`  close_gate: enforce` — hook ENFORCES, reader says report-only (split-brain: commits blocked while
state.mjs/docs-audit stay report-only); (b) a glued comment `close_gate: enforce#note` — reader ENFORCES
(`[^\s#]+` stops at `#`), hook says report-only (`enforce([[:space:]]|$)` fails); (c) duplicate keys —
reader takes the first occurrence, grep matches any line. `test_031_guard_config_conformance`
(tests/skills/test-aai-hygiene-pack.sh) covers absent/enforce/report-only/trailing-comment/commented-out/
invalid but NOT indented, glued-comment, duplicate-key, quoted-value, or CRLF variants (quoted and CRLF
happen to agree; the other three do not), so D8's "drift now fails a test" promise does not hold for these
classes. **Fix:** anchor the hook greps to column 0 (drop `[[:space:]]*` after `^`) — the reader's
documented "an indented or commented key is never a dial" semantics is the canonical one — and add the
missing variants to test_031 (which would then have caught this).

**W3 — flush resume path silently discards the metrics of a legitimately RE-OPENED ref.**
`metrics-flush.mjs:570`: any STATE metrics entry whose ref is already in the ledger is classified as an
interrupted flush and given a cleanup-only pass. Failure scenario: a CHANGE is re-opened after a past
flush (same ref_id, new active_work_items entry, new agent_runs, fresh validation PASS); the next flush
removes the new entry from STATE without appending anything — the second cycle's telemetry is
unrecoverable (the report's `RESUME <ref>` line is the only trace, and it mislabels what happened). The
old prompt had the adjacent flaw (the ref was simply never flushable again), so this is not a regression,
but the script now actively deletes the data. **Disposition:** follow-up ref — either refuse the resume
when the STATE entry contains runs newer than the ledger entry's `date_utc` (surface human_input), or
document that re-opened work must take a new ref id. Related note (accept as-is): two concurrent flush
processes could both append before either commits STATE — the second one's `writeState` recheck aborts it
but its duplicate ledger line survives; single-writer posture (SPEC-0012 W4) is the documented guard.

**W4 — flush criterion `refMatches(last_validation.ref_id, ref)` is stricter than both the prompt and D6, and strands the two done items in the real STATE today.**
`metrics-flush.mjs:571`. Old prompt criteria: "validation verdict is PASS" (one shared block); spec D6:
"validation PASS/CANCELLED" — neither requires the verdict to NAME the ref. The addition is a sound
staleness guard (aligned with dispatch rule 14's vmatch), but its interaction with full reset creates a
permanent trap: the real STATE currently has CHANGE-0010 and CHANGE-0011 done with unflushed metrics
entries while `last_validation` names CHANGE-0009. When CHANGE-0009 flushes, `remaining.every(done)`
(line 607) triggers a FULL reset (validation -> not_run, focus cleared) and ephemeral cleanup (LOOP_TICKS
deleted), after which no future `last_validation` can ever name 0010/0011 — their entries are unflushable
by the script forever (skip reason: "validation verdict does not name <ref>"). The skip reasons are
visible, so nothing is silent, but the telemetry needs an operator decision BEFORE the first real flush:
either flush them by hand-appending per .aai/STATE_FALLBACK.md and removing the entries, or record a
decision to drop them, or sequence per-ref flushes with a pass verdict naming each. **Disposition:**
decision/follow-up ref required at closeout.

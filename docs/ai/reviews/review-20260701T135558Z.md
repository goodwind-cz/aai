## Stage 2 — Code Quality

### WARNING

**W1 — Pre-existing residual git-volatile call in committed-INDEX path**
`generate-docs-index.mjs:207` — `firstCommitDate(ROOT, rel)` is called inside the legacy-autoskip path, ONLY when `loadConfig(ROOT)?.legacy_until_date` is non-null AND there are whole-doc schema failures. When both conditions hold, the decision of which failing docs are "legacy-autoskipped" (excluded from the committed index silently) depends on git history. This means the committed INDEX is NOT a pure function of on-disk docs under that configuration.

The PR correctly removed the `runAudit()`-scoped sections (Spec-AC-02 as literally written is satisfied — the concrete test mutates git history by adding commits that reference a doc ID, and `firstCommitDate` is immune to that). However, the broader design intent ("committed index is a pure function of on-disk docs") is violated when `legacy_until_date` is configured.

**Pre-existing:** This code predates SPEC-0010 and was not introduced by this PR.
**Dormant:** The current repo has no `legacy_until_date` config; `legacyUntil` is always null.
**Severity:** WARNING (not a regression, but a latent spec-scope gap that the PR's comments claim to fix).

**Suggested fix (non-blocking for merge):** Move the `firstCommitDate` call from the committed-INDEX path to the companion / violations report only, or document this known gap explicitly in SPEC-0010 as a deferred item.

### INFO

**I1 — `--repair` cannot handle `metrics: {}` (inline empty mapping) form**
`metricsBlockRanges` uses `/^metrics:\s*$/` which requires a bare mapping header. A `metrics: {}` (inline empty mapping) is correctly detected as a duplicate key by `topLevelKeyCounts`, but `--repair` fails to find the block (ranges length stays at 1 from the normal block), writes nothing, then re-validates and exits 1 with a clear error. The user is told there's still a duplicate but the repair message doesn't clearly explain why repair couldn't merge the inline form.

Practically unreachable: real STATE.yaml always uses `metrics:` as a block mapping header. Detection and fail-loud behavior are both correct.

File: `.aai/scripts/check-state.mjs` lines 55-68

**I2 — No atomic write in `repairMetrics`**
`fs.writeFileSync(abs, ...)` is not atomic (no temp-file-then-rename). A process crash during the write could leave STATE.yaml partially written.

Acceptable for a dev-tool validator. Mitigation: the state is only written after the merge computation is complete (single `writeFileSync` call). The risk window is small.

File: `.aai/scripts/check-state.mjs` line 194

**I3 — `2>/dev/null || true` silently swallows unexpected git rm errors**
`.aai/scripts/install-pre-commit-hook.sh:94` and `.ps1` equivalent:
`git rm --cached --quiet --ignore-unmatch docs/INDEX.audit.md 2>/dev/null || true`
The `2>/dev/null` suppresses ALL stderr, including unexpected errors (disk full, permissions). `--ignore-unmatch` handles the normal "file not in index" case, so `2>/dev/null` is redundant and masks real errors.

Low risk: the hook continues either way and the `--ignore-unmatch` flag already handles the expected error case.

**I4 — Working tree INDEX.md was stale at start of review**
The working tree `docs/INDEX.md` was generated at 13:36:55 before the validation agent updated SPEC-0010's AC rows and frontmatter to `done` at ~13:37. This left the INDEX stale (showing SPEC-0010 as `implementing`) and caused `test_issue0001_posix_paths_noop` (TEST-003/SPEC-0007) to fail, blocking the SPEC-0010 test group from running until the INDEX was refreshed. A follow-up would be to ensure the pre-commit hook or the final validation step always regenerates the INDEX after updating doc statuses.

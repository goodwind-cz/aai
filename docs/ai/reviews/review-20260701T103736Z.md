## E1 Disposition — Independently Confirmed FIXED

### What E1 was

Prior review found the workspace guard:
```sh
case "$rest" in
  *"$WORKSPACE"*) ;;
  *) continue ;;
esac
```
allowed a workspace path that is a string prefix of a sibling (e.g., `/x/myproject` is a prefix of `/x/myproject-fork`) to match the sibling's process, causing an over-kill. Independently demonstrated by live execution: reaper scoped to `WORKSPACE=/tmp/.../myproject` killed a process in `/tmp/.../myproject-fork` (reaped: 1).

### Remediation applied

`.aai/scripts/aai-reap-tests.sh` lines 35 and 111:
```sh
WORKSPACE="${WORKSPACE%/}"    # strip trailing slash before guard
...
[[ "$rest" == *"${WORKSPACE}/"* ]] || continue
```

### Independent repro: before/after

Live execution with WORKSPACE=`/tmp/e1_verify.rrIg06/myproject`:

```
In-workspace process:     vitest_/tmp/e1_verify.rrIg06/myproject/worker
Prefix-sibling process:   vitest_/tmp/e1_verify.rrIg06/myproject-fork/worker

OLD guard *"$WORKSPACE"*:         MATCHES fork process => WOULD OVER-KILL (confirmed E1)
NEW guard *"${WORKSPACE}/"*:      does NOT match fork process => CORRECT, E1 fixed

Reaper output: reaped: 1
In-workspace match_pid alive after reap: NO  (CORRECT — was reaped)
Prefix-sibling fork_pid alive after reap: YES (CORRECT — spared)
```

E1 is **genuinely resolved**. The path-separator guard requires a literal `/` immediately after the workspace path, so `/x/myproject` cannot match `/x/myproject-fork`.

### Trailing-slash strip correctness

`WORKSPACE="${WORKSPACE%/}"` strips exactly one trailing slash. If WORKSPACE were `/x/project/`, the guard `*"/x/project/"*` would still be correct (the stripped WORKSPACE `/x/project` + `/` reconstructs `/x/project/`). If WORKSPACE has no trailing slash (the common case), `%/` is a no-op. Correct.

### Glob-safety of the new guard (I3 from prior review also resolved)

`[[ "$rest" == *"${WORKSPACE}/"* ]]` — in bash, the double-quoted `"${WORKSPACE}/"` portion is treated as a literal string, not a pattern. So `*`, `?`, `[`, `]` inside WORKSPACE are matched literally. The surrounding unquoted `*` remain pattern wildcards. Glob-metacharacter-safety is confirmed. I3 is resolved.

## Stage 2 — Code Quality Findings

### WARNING W1 — Portability regression: `#!/bin/sh` + `[[ == ]]` (bash-only) (NEW)

**File:** `.aai/scripts/aai-reap-tests.sh`
**Line:** 111
**Category:** Portability / Spec compliance

**Finding:**

The remediation changed the workspace guard from a POSIX `case` pattern to a bash-specific `[[ == ]]` construct:

```sh
[[ "$rest" == *"${WORKSPACE}/"* ]] || continue
```

`[[ ]]` is a bash extension and is NOT available in POSIX sh (dash). The script header is `#!/bin/sh` and its comment says "POSIX sh; works on macOS (BSD ps) + Linux (GNU ps)." On systems where `/bin/sh` is dash (Ubuntu 22.04, Debian, Alpine, and many CI containers), `[[` is not recognized as a builtin:

```
$ dash -c '[[ "x" == *"y"* ]] || echo "no match"'
dash: 1: [[: not found
no match
```

In the while loop of the reaper, `[[ ... ]] || continue` behaves as: `[[` fails (command not found), `|| continue` always fires, every iteration is skipped. The reaper becomes a **silent no-op** on dash: it exits 0 and prints `reaped: 0`, never killing any process.

This does NOT create an over-kill path (safe direction — fails to reap, does not kill wrongly), but it violates the spec's portability claim and constitutes a meaningful false-negative regression for Linux deployments where `/bin/sh` is dash.

**POSIX sh alternative (same semantics, fully portable):**
```sh
# POSIX sh case with quoted pattern — identical semantics to [[ == ]]:
# The quoted "${WORKSPACE}/" portion is matched literally (not as a glob) in case.
case "$rest" in
  *"${WORKSPACE}/"*) ;;
  *) continue ;;
esac
```

This achieves:
1. Trailing-slash path-separator anchoring (fixes E1)
2. Glob-metacharacter safety (POSIX sh quoting in case patterns suppresses glob expansion)
3. POSIX sh portability (works with dash, bash 3.2+, ksh, etc.)

**Why this is WARNING, not ERROR:**
- On the primary target platform (macOS, where `/bin/sh` is bash 3.2), the reaper works correctly.
- The TEST suite passes 12/12 on macOS.
- No over-kill risk is introduced.
- The regression is in the POSIX sh portability claim, not in safety.

**Should fix before merge** to maintain spec compliance ("POSIX sh; works on ... Linux") and prevent silent no-op on CI containers / Linux hosts.

### INFO I5 — Narrow false-negative for `--root /path` without trailing slash (pre-existing, unchanged by remediation)

**File:** `.aai/scripts/aai-reap-tests.sh`
**Lines:** 111
**Category:** Correctness (safe direction)

The new guard `*"${WORKSPACE}/"*` requires a literal `/` immediately after the workspace path. A real vitest invocation using `--root /x/myproject` (without trailing slash) would have `/x/myproject` in its argv but NOT `/x/myproject/`. The old guard `*"$WORKSPACE"*` would have matched this; the new guard does not (false negative introduced).

**Practical impact:** Low. In the common case, vitest processes launched inside a workspace have the workspace path embedded in the full path to the vitest binary in `node_modules`:
```
node /x/myproject/node_modules/vitest/dist/cli.js run
```
which DOES contain `/x/myproject/` (with trailing slash). The `--root /x/myproject` (no slash) scenario is a minority invocation style.

**Assessment:** Safe-direction false negative. The spec uses this as "defence-in-depth" (D3: "defence-in-depth reaper"). Missed leaks from the `--root` form degrade defence-in-depth but don't cause over-kill. INFO only.

## Analysis of TEST-005/006 Marker Changes

Markers changed from `vitest_run_${ws}_worker` to `vitest_run_${ws}/worker`.

**Why the change was necessary:** With the new guard `*"${WORKSPACE}/"*`, the old marker `vitest_run_${ws}_worker` would NOT match (the `_worker` suffix doesn't contain `/`), causing TEST-005's positive control (the process that SHOULD be reaped) to survive — falsely failing the reaper's in-workspace kill. The marker change is required for the positive control to work with the new guard.

**Did the change weaken the tests?**

No. The test still exercises the correct invariant:
- **Positive control** (marker `vitest_run_${ws}/worker`): workspace path embedded as a proper path component (with `/`). This simulates a real process like `node /ws/node_modules/vitest/...`. Guard matches → process is reaped. Correct.
- **Negative control in TEST-005** (marker `vitest_run_${other}/worker`): `other` is a completely different mktemp directory (`other.XXXX` vs `ws.XXXX`). Guard does NOT match → process survives. Correct.
- **Negative control in TEST-006** (fresh `vitest_fresh_${ws}/worker`): age guard protects it. Guard matches but etime is below threshold → process survives. Correct.

The E1-specific scenario (workspace is a PREFIX of sibling) is tested by TEST-012, not TEST-005/006. The separation is clean.

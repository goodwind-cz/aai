## Stage 2 — Code quality findings

### WARNING [BP-001] `gateDoc()` does not pass `extraMethods` to `parseReviewBy` — diverges from `runAudit` under `review_by_methods` config
`.aai/scripts/lib/docs-audit-core.mjs:619`

`gateDoc` calls `parseReviewBy(row['Review-By'])` with no second argument, whereas `runAudit` calls `parseReviewBy(row['Review-By'], extraMethods)` where `extraMethods = config.review_by_methods` (`docs-audit-core.mjs:283,352,365`). `extraMethods` only affects the `combo` (`method:date`) and `actor-method` (`actor method`) forms (`docs-model.mjs:112-119,126-131`); bare labels use the fixed `REVIEW_BY_LABELS` whitelist regardless. Consequence: in a downstream repo that configures a custom `review_by_methods` entry and uses it in a `combo`/`actor-method` Review-By token, `gateDoc` classifies that token `kind: 'invalid'` and fails the gate, while the full audit accepts it. Under `close_gate: enforce` this is a false gate FAIL that refuses a legitimate done-flip or aborts a commit.

Not triggered in this repo: `review_by_methods` is empty (default `[]`), so there is no live divergence and no exit-code/behavior impact here. Fix is one line: load the config in `gateDoc` (it already calls `loadConfig`) and pass `config.review_by_methods ?? []` into `parseReviewBy`.

### INFO [BP-002] G3 artifact corroboration uses substring `name.includes(id)` with no id boundary
`.aai/scripts/lib/docs-audit-core.mjs:143`

`reviewArtifactExists` matches an artifact by `name.includes(id)`. `DOC_ID_RE` permits 1–5 digit numbers (`\d{1,5}`), so with mixed-width ids a shorter id (e.g. `SPEC-001`) is a substring of a longer sibling's artifact filename (e.g. `review-SPEC-0011.md`), yielding a false-positive corroboration. The event-side roll-up (`refMatch`, `docs-audit-core.mjs:146` and the G2 check at `:379`) is correctly boundary-guarded (`=== id || startsWith(id + '/')`); only the artifact glob is loose. No live impact: this repo's ids are uniformly 4-digit (the SPEC-0005 vs SPEC-0050 pair the spec calls out does not bleed), and the check is report-only. Matches the spec's stated residual risk (design decision G3).

### INFO [BP-003] Real review artifacts are timestamp-named, so the G3 artifact-glob path is effectively dead in this repo
`docs/ai/reviews/`

The existing review files are named `review-<timestamp>.md` (e.g. `review-20260703T003739Z.md`) and contain no doc id, so `reviewArtifactExists` never fires for them; G3 corroboration in practice depends entirely on `code_review_completed` / `work_item_closed` events. This is consistent with the spec's recorded residual risk (report-only + triple corroboration source + positive-control test) and is not a defect, but the artifact-glob branch is untested against the actual naming convention.

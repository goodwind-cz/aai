## Purpose

This is an adversarial independent re-review of the E1 remediation. The prior
review FAILED the implementation because the expired-lock self-heal did an
unconditional `rmSync` by path, letting two concurrent acquirers observing one
expired lock both exit 0 (double-claim). The remediation serializes the expired-
reclaim through a short-lived O_EXCL `.reclaim` sentinel and re-verifies expiry
under the sentinel. This re-review must decide: does the sentinel fix actually
close Spec-AC-03, or does it introduce a new race?

---

## STAGE 1 — Adversarial Race Analysis (sentinel reclaim logic)

### 1a — SAFE: loser at sentinel EEXIST spins, never acts on p

**Verdict: SAFE.**

When two acquirers both see the expired lock (openSync EEXIST, readLock -> expired),
they race for the sentinel (`openSync(sentinelPath, 'wx')`). Exactly one wins (O_EXCL).
The loser gets EEXIST on the sentinel. The loser's code path is:

```js
// EEXIST on sentinel
if (sentinelIsStale(sentinelPath)) {
  try { fs.rmSync(sentinelPath, { force: true }); } catch { /* gone */ }
}
sleepMs(2);
continue;   // <-- back to the top of the for-loop
```

The loser does NOT call rmSync(p) and does NOT call openSync(p, 'wx'). It sleeps
2ms and re-enters the for-loop, where it tries the fast-path openSync(p, 'wx') again.
If p still exists (sentinel holder hasn't acted yet), openSync gets EEXIST again and
the loser re-tries the sentinel (spinning). The loser NEVER proceeds past the sentinel
check to remove or create p while another process holds the sentinel. Correct.

### 1b — SAFE: fast-path competitor vs sentinel holder; O_EXCL re-arbitrates the empty slot

**Verdict: SAFE.**

After the sentinel holder calls `rmSync(p)` (deletes the expired lock), the slot
is briefly empty. A fast-path acquirer (e.g., one that was spin-retrying at the
top of the loop when p disappeared) can call `openSync(p, 'wx')` and win the
O_EXCL create. If that happens BEFORE the sentinel holder calls its own
`openSync(p, 'wx')`, the sentinel holder gets EEXIST on its create:

```js
try {
  const nfd = fs.openSync(p, 'wx');     // sentinel holder's O_EXCL create
  ...
  outcome = { ok: true };
} catch (ce) {
  if (ce.code !== 'EEXIST') throw ce;
  outcome = { held: (readLock(p) || {}).owner };  // sentinel holder loses -> 3
}
```

The sentinel holder correctly handles EEXIST by setting `outcome = { held: ... }`,
releasing the sentinel in the `finally` block, then calling `fail(..., 3)` (AFTER
the finally). It does NOT attempt a second create. The O_EXCL guarantees exactly
one of {sentinel holder, fast-path competitor} creates p. Two creates cannot both
succeed. Correct.

### 1c — SAFE: fresh lock is never rmSync'd by the sentinel holder

**Verdict: SAFE.**

The sentinel holder re-reads p under the sentinel before any delete:

```js
const cur = readLock(p);
if (cur && !isExpired(cur)) {
  outcome = { held: cur.owner };   // fresh lock: do NOT delete, exit 3
} else {
  if (cur) { try { fs.rmSync(p, { force: true }); } catch { /* gone */ } }
  // ... O_EXCL create ...
}
```

`rmSync(p)` is only called when `!cur || isExpired(cur)`:
- If `cur` is null (p is gone — a previous sentinel holder already deleted it):
  `if (cur)` is false, no rmSync called. The holder proceeds directly to O_EXCL create.
- If `cur` is an expired lock: rmSync deletes it. Could another process have replaced
  it with a fresh lock between `readLock(p)` and `rmSync(p)`? No: the sentinel
  exclusively gates all other reclaimers (they spin at sentinel EEXIST). A fast-path
  acquirer could only create a fresh lock at p if p were empty; but p is non-empty
  (holds the expired lock) at the time of readLock. For a fast-path acquirer to win
  p, it would have to see `openSync(p,'wx')` succeed, which requires p to not exist
  — impossible while p has an expired lock. Therefore no process other than the
  sentinel holder can replace p's expired lock with a fresh one in that window.

The re-read-then-rm window is safe under the sentinel. Correct.

### 1d — SAFE (with accepted theoretical bound): stale-sentinel stealing re-arbitrates

**Verdict: SAFE in practice (theoretical edge accepted per spec).**

If two acquirers simultaneously observe a stale sentinel (age > SENTINEL_STALE_MS
= 30,000ms), both call `rmSync(sentinelPath, { force: true })`. One succeeds; the
other silently swallows the ENOENT (force:true). Both then `sleepMs(2); continue`.
On the next iteration, both re-try `openSync(p, 'wx')`:

- If p still has an expired lock: both get EEXIST, both try `openSync(sentinelPath,
  'wx')` — one wins via O_EXCL. The other spins. Correct.
- If p is empty (the stale sentinel's dead holder had already deleted p): both
  race `openSync(p, 'wx')` — exactly ONE wins via O_EXCL. Correct.

Could one steal-and-delete the fresh sentinel that the other created? No: by the
time a stealer calls `rmSync(sentinelPath)` they have already called
`sentinelIsStale(sp)` which returned true (> 30s old). A sentinel created in the
last millisecond is NOT stale; `sentinelIsStale` returns false, preventing the
steal. Therefore a fresh sentinel is never deleted by a stale-steal. Correct.

Theoretical edge: if a sentinel holder's process is suspended by the OS for 30+
seconds while inside the critical section, a stealer could delete its sentinel file,
win a new sentinel, and both could hold the critical section simultaneously. This
requires OS-level suspension for 30s during filesystem syscalls — implausible under
normal operation. The spec explicitly accepts this as the SENTINEL_STALE_MS
operational boundary ("a real reclaim is a few syscalls; a sentinel older than this
bound must be a leak from a dead acquirer"). Accepted per spec; not a regression.

### 1e — SAFE: process.exit is never called while holding the sentinel

**Verdict: SAFE.**

The code uses the `outcome` variable pattern so control flow that triggers
`process.exit` (via `fail()`) is always OUTSIDE the `try { ... } finally { release }`:

```js
let outcome = null;
try {
  // ... all sentinel-critical work; only sets outcome, never calls fail() ...
} finally {
  fs.closeSync(sfd);                              // release sentinel fd
  try { fs.rmSync(sentinelPath, { force: true }); } catch { /* gone */ }  // delete sentinel
}
// Sentinel fully released before this line
if (outcome.ok) { console.log(...); return; }    // normal return
fail(`... held by ...`, 3);                       // process.exit(3) AFTER finally
```

The `fail()` call is reached only after the `finally` block has run. All throws
from inside the try block (e.g., non-EEXIST errors from `openSync(p,'wx')`) also
trigger the `finally` before propagating. The sentinel is always released. Correct.

### 1f — ACCEPTABLE: sentinel leak on crash recovers within SENTINEL_STALE_MS

**Verdict: ACCEPTABLE per spec.**

If an acquirer crashes while holding the sentinel (sfd open, sentinel file exists,
mtime set by the best-effort writeSync), other acquirers see EEXIST on the sentinel
but detect it is stale after 30 seconds. They steal and delete it (sentinelIsStale
check + rmSync), then compete normally. Additionally, `reap` sweeps stale `.lock.reclaim`
files (verified empirically: reap removes stale sentinels, leaves fresh ones, removes
expired locks — all correct). Wedge duration is bounded by SENTINEL_STALE_MS (30s) or
next `reap` call. Spec-D4 explicitly records this as residual risk R-TTL; the 30s
bound is well within the operational TTL of 1800s. Acceptable per spec design.

## Stage 2 — Code Quality Update

No new ERROR or WARNING findings introduced by the sentinel fix. The sentinel
logic is well-structured: O_EXCL for the sentinel itself (same primitive as the
lock); `try { ... } finally { release }` pattern that is process.exit-safe; bounded
spin (MAX_RECLAIM_ATTEMPTS=64); `sleepMs` using Atomics.wait (no busy-spin);
stale-detection via mtime. The added code follows the same node:fs-only convention
as the rest of the file.

New observation (no action required): the `sleepMs(2)` is called unconditionally
after the stale-steal rmSync, which means two rapid stealers both sleep 2ms before
re-competing. This is a minor liveness cost but prevents a thundering-herd on
a single stale sentinel and is intentional.

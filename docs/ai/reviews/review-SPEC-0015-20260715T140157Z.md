2. The validation report carries `type: validation-report` (not in
   `DOC_TYPE_ENUM`) and `status: done` with no close telemetry/commit, so the
   drift engine classes it `probable-false-done` and emits a type warning.

Why it's real, not cosmetic: the SPEC-0015 GREEN evidence log
(`green-20260715T133821Z.log`, 13/13) was captured BEFORE this report existed;
the validation run reported "CLEAN" over 37 docs. The delivered tree now has 38

test is red right now. `docs/validation/` is neither gitignored nor in

`bash tests/skills/test-aai-doc-numbering.sh` to confirm TEST-013 green and
`docs-audit` CLEAN.

### CR-2 — NON-BLOCKING (INFO) — positional-arg parse is a dead no-op

`allocate-doc-number.mjs:279`
`opts._extra = (opts._extra || []).push?.(tok);` — `Array.prototype.push`
returns the new length (a number), so `_extra` becomes a number after the first
positional token and `.push?.` on a number is `undefined` thereafter. `_extra` is
never read anywhere, so a stray positional argument is silently swallowed rather
than rejected. No functional impact (the CLI contract is flag-only); worth a
one-line cleanup (drop `_extra` or make bare positionals a usage error → exit 2)
for consistency with the strict `_badFlag` handling two lines above.

### CR-3 — NON-BLOCKING (INFO) — batch allocation is validate-then-write, not transactional across fs failures

`runAllocate()` correctly validates ALL drafts (slug, prefix, base-ref
collision) in the plan phase before any write, so a *guard* failure yields no
partial rename (matches D3/TEST-005). The subsequent write loop
(allocate-doc-number.mjs:434-443) stamps+moves+rewrites per draft sequentially; a
mid-loop *filesystem* error (e.g. EACCES on draft 2 of an `--all` batch) would
leave draft 1 already renamed. This is an accepted edge (single-draft `/aai-pr`
is the normal path; git makes it recoverable) — noting for completeness, not a
required fix.

## Non-defects verified (spent effort, found clean)

- `baseRefNumbers`/`nextNumber`: first-doc case → RFC-0001 (max 0 + 1); width-4
  zero-pad correct; `numberFromBasename` lookahead `(?=[-.])` correctly parses
  `RFC-0007-slug.md`, `RFC-0007.md`, and rejects `RFC-DRAFT-*`.
- Type-prefix vs directory mapping: `docs/issues` hosting ISSUE/CHANGE/DEBT is
  handled by reading the DISPLAY prefix from the filename leading token
  (`prefixFromBasename`), so ISSUE and CHANGE keep independent number sequences
  and never false-collide in the duplicate guard. Compound prefixes
  (DECISION-RFC, SPEC-CHANGE) resolve consistently.
- Both guards on legacy docs: no-DRAFT predicate keys on `'number' in fm` (a
  legacy doc WITHOUT a `number` field is never flagged); duplicate predicate
  skips prefix-less files. Ran `--guard` on the real repo → exit 0, clean. No
  false positives.
- `git mv` + stamp + reference-rewrite: `stampNumber` replaces `number: null`
  in-place preserving all other bytes; `rewriteReferences` keys on the full
  `TYPE-DRAFT-<slug>` basename (specific, low false-hit risk) and only touches
  governed dirs; CRLF handled upstream by `normalizeNewlines`; backfill idempotent.
- Exit-code discipline: 0/2/3/4 exact; exit 3 leaves DRAFT byte-identical
  (TEST-005 verified); degrade-and-report when git/node/allocator absent (host
  emits a pass line, intake/PR fall back).
- Injection/quoting: shell guard invokes a fixed script path, captures output,
  pipes through `sed` — no eval, no interpolation of untrusted data into a shell.
  `.ps1` mirror matches `.sh` behavior (same default report-only, same
  `doc_number_guard: enforce` flip regex, same skip-when-absent).
- Backward compatibility: existing numbered docs without a `number` field still
  render via the `fm.id ?? basename` fallback (generate-docs-index.mjs:213-217);
  index byte-idempotent modulo the Generated line (TEST-009 + live re-run).

## Guardrail assessment (the central question)

**Genuinely narrowed, not removed.** The replacement text bounds the resolver
along four independent axes:

1. Exactly one row (`the resolver may act on EXACTLY the one row matched`,
   :84-85).
2. Exactly one field (`PLUS the ONE declared target field`, :138-140).
3. Exactly one channel (`via the typed state.mjs CLI — nothing else`, plus
   `Never hand-edit docs/ai/STATE.yaml`).
4. A closed vocabulary (`never invent a setter or flag that the mapping table
   does not name`), repeated verbatim in STRICT RULES at :169-172 so a resolver
   that skims to the rules block still hits it.

`none` rows are an asserted verdict, not an omission (`a trigger whose target is
`none` permits NO STATE write beyond `human_input``). I could find no reading
under which a resolver concludes it may touch `last_validation`,
`current_focus`, `locks`, `tdd_cycle`, or `active_work_items`. The residual
ambiguity in this diff runs the *other* way — the guardrail is tight enough to
also block the `human_input` clear it is supposed to permit (finding 1).

## Write ordering and fail-closed

Ordering is stated twice and unambiguously: `run the target setter BEFORE
clearing human_input in STEP 5` (:125-126) with the crash rationale, echoed in
STEP 5's own lead-in (`target setter from STEP 4c first`). The non-zero-exit rule
is explicit: `If the setter exits non-zero, do NOT clear human_input; report the
exit code and STOP — the block stays raised` (:123-124). Both are prose-only
obligations with no automated coverage (see cannot_verify).

Fail-closed forbids guessing and defaulting explicitly (`MUST NOT guess, MUST NOT
pick a default, and MUST NOT clear human_input`), with a bounded one-follow-up
budget and a distinct `HITL UNRESOLVED` output block. I found no path where a
resolver picks a default silently. The one wording defect is over-breadth, not
under-breadth (finding 2).

## 1. safeScope() Analysis

### Change Summary

Old:
```js
function safeScope(scope) {
  return String(scope).replace(/[^A-Za-z0-9._-]/g, '_');
}
```

New:
```js
const FS_SAFE_RE = /^[A-Za-z0-9._-]+$/;

function safeScope(scope) {
  const raw = String(scope);
  if (FS_SAFE_RE.test(raw)) return raw;
  const sanitized = raw.replace(/[^A-Za-z0-9._-]/g, '_');
  const hash = crypto.createHash('sha256').update(raw).digest('hex').slice(0, 8);
  return `${sanitized}-${hash}`;
}
```

### a. Back-compat — SAFE

A raw scope already in `[A-Za-z0-9._-]+` is returned verbatim. The old code applied `.replace(/[^A-Za-z0-9._-]/g, '_')` to such inputs, which is a no-op (no chars replaced). Both old and new return the same value for any FS-safe scope.

CLI confirmation:
```
$ AAI_LOCK_DIR=/tmp/t node docs-lock.mjs acquire CONTESTED owner1
acquired CONTESTED -> owner1 (ttl 1800s, pid 59867)
$ ls /tmp/t
CONTESTED.lock     # identical to pre-patch behaviour
```

Pre-existing scopes (CONTESTED, RFC-0001, docs-lock, etc.) are unaffected.

### b. Determinism — SAFE

SHA-256 is a pure function of the raw string. No `Date`, `Math.random`, or process-state in the suffix. The function is referentially transparent: same raw string always yields the same output. No non-determinism found.

### c. Injectivity — FINDING (LOW severity)

**Same-branch (both safe)**: SAFE. Each safe scope maps to itself; distinct safe scopes produce distinct outputs.

**Same-branch (both unsafe)**: SAFE. Two distinct unsafe scopes with identical sanitized stems are disambiguated by their SHA-256 hash, which has a collision probability of 1/16^8 ≈ 2.3×10⁻¹⁰ per pair — negligible.

**Cross-branch (one safe, one unsafe)**: INJECTIVITY VIOLATION FOUND.

The unsafe path produces output `${sanitized}-${hash}` where the suffix contains only `[0-9a-f]` and the separator is `-`. This output is itself a valid FS-safe string (all chars in `[A-Za-z0-9._-]`). Therefore, a safe scope whose literal name equals the output of an unsafe scope's safeScope() call maps to the same lock filename.

Demonstrated collision pair:
- Scope `"a/b"` (unsafe) → `safeScope` → `"a_b-c14cddc0"` → lock file `a_b-c14cddc0.lock`
- Scope `"a_b-c14cddc0"` (safe, passes FS_SAFE_RE) → `safeScope` → `"a_b-c14cddc0"` → lock file `a_b-c14cddc0.lock`

CLI confirmation:
```
$ AAI_LOCK_DIR=/tmp/t node docs-lock.mjs acquire a/b owner-ab
acquired a/b -> owner-ab (ttl 1800s, ...)
$ AAI_LOCK_DIR=/tmp/t node docs-lock.mjs acquire a_b-c14cddc0 owner-crafted
docs-lock: scope "a_b-c14cddc0" is held by owner-ab (not expired)
exit=3
$ ls /tmp/t
a_b-c14cddc0.lock     # only 1 file; 2 expected for distinct scopes
```

**Severity assessment — LOW / non-blocking:**

- Triggering the collision requires a scope literally named with the `<sanitized>-<hex8>` pattern of another scope (e.g., `a_b-c14cddc0`). Real scope names used in this codebase are human-readable identifiers (`RFC-0001`, `CONTESTED`, `docs/auth`, `SPEC-0004`). None of these would accidentally match the hash-derived output of another scope.
- No security boundary at stake: single-tenant dev tooling.
- The fix correctly solves the intended bug (same-sanitization, different raw → distinct files); the residual case requires deliberate construction of the scope name.

**Recommendation (non-blocking):** Namespace unsafe outputs to prevent the cross-branch collision — e.g., always prepend a sigil like `~` to unsanitized scope outputs (`~a_b-c14cddc0`) so they can never collide with naturally-safe scopes. Alternatively, document the invariant that scope names must not match the pattern `*-[0-9a-f]{8}$`.

### d. Path Safety — SAFE

For unsafe scopes, all `/`, `\`, and other path-separator characters are replaced by `_` before hashing. The output contains only `[A-Za-z0-9._-]` plus the hash suffix (also only `[0-9a-f-]`). No path separator survives into `lockPath()`:

```js
function lockPath(scope) {
  return path.join(LOCKS_DIR, `${safeScope(scope)}.lock`);
}
```

Edge cases verified:
- `..` → safeScope passes FS_SAFE_RE → returns `..` → `path.join(LOCKS_DIR, "...lock")` — file named `...lock` in LOCKS_DIR, not a traversal.
- `.` → `path.join(LOCKS_DIR, ".lock")` — hidden file in LOCKS_DIR, not a traversal.
- `../etc/passwd` → sanitized to `_._etc_passwd` → lockfile `_._etc_passwd-HASH.lock` in LOCKS_DIR.
- `long scope (500+ chars)/b` → output length 511, no slash.

All outputs stay within LOCKS_DIR.

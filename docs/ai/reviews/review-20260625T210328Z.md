## Stage 2 — Code Quality findings

### WARNING

**[WARN-1] Reverse-link union path is never independently exercised by the tests** — `tests/skills/test-aai-docs-audit.sh` (closeout fixtures, ~lines 690-840)
Spec-AC-01 promises resolution is the *union* of forward `asList(links.spec)` and reverse `links.rfc`/`links.requirement`. The code implements this correctly (core.mjs:383-385). However, in every fixture the reverse link points to the *same* spec id already present in the parent's forward `links.spec` (RFC-0010↔SPEC-0010, RFC-0020↔SPEC-0021/0022, etc.). No fixture has a parent that resolves a spec *only* via the reverse path (parent without `links.spec`, child spec with `links.rfc` back-pointer). Consequently, deleting the entire reverse-resolution block would not fail any test — the union branch is untested.
Recommendation (non-blocking; the code is correct by reading): add a reverse-only fixture — e.g. an `rfc` parent (proposed) with NO `links.spec`, plus a `done` spec whose `links.rfc` names that parent — and assert the parent IS flagged. This makes the AC-01 "union" claim a real RED.

### INFO

**[INFO-1] Forward `links.spec` resolution does not assert the target is `type: spec`** — `.aai/scripts/lib/docs-audit-core.mjs:387`
The reverse path filters `docType(d) === 'spec'`, but the forward path counts any `links.spec` id whose resolved doc has `status: done`, regardless of its actual type. A misfiled `links.spec` pointing to a non-spec done doc would count as a satisfying "spec" and could flag the parent. Low real-world risk (`links.spec` is operator-declared). For symmetry/robustness consider `r && r.status === 'done' && docType(r) === 'spec'` in the all-done check.

**[INFO-2] Reverse resolution is O(parents × docs)** — `.aai/scripts/lib/docs-audit-core.mjs:383-385`
Each candidate parent rescans all docs for back-links. Negligible at current repo scale and avoids extra state, but a single prebuilt `parentId -> [spec ids]` reverse index would make it O(docs). Optional.

**[INFO-3] `counts.closeoutCandidates` is populated but not surfaced in the summary line** — `.aai/scripts/lib/docs-audit-core.mjs` (counts) / `.aai/scripts/docs-audit.mjs:77-78`
The integer count is added to `counts` and returned, but the digest summary header (Scanned/Orphans/Drifted/...) does not include it; only the dedicated section shows the count. Harmless and arguably correct (keeps it out of triage framing), noted for awareness.

**[INFO-4] `type: requirement` parents are out of scope by design** — `.aai/scripts/lib/docs-audit-core.mjs:30` (`CLOSEOUT_PARENT_TYPES = {rfc, prd}`)
`DOC_TYPE_ENUM` distinguishes `prd` from `requirement`. The scope intentionally uses `prd` per CHANGE-0004; a PRD-equivalent authored with `type: requirement` would not be flagged. This matches the spec (false-positive guard) — flagged only so the scope decision is visible.

## Stage 2 — Code Quality

### WARNING-1 — Orphans section path not POSIX-normalized on Windows

**Category:** Best practices / OS portability  
**File:** `generate-docs-index.mjs:361` and `docs-audit-core.mjs:136`

The Orphans section of `docs/INDEX.md` renders `d.rel` for each orphan doc:

```js
// generate-docs-index.mjs ~line 361
for (const d of items) out.push(`| ${d.rel} | ${d.legacy ? ...} | ${d.reasons.join('; ')} |`);
```

`d.rel` originates from `scanAuditDocs()` in `docs-audit-core.mjs:136`:

```js
const rel = path.relative(root, full);  // no .split(path.sep).join('/')
```

This path is NOT POSIX-normalized. On Windows, `path.relative()` returns backslash-separated paths, and the Orphans section of `docs/INDEX.md` would contain backslash paths, contradicting the spec's Design decision 2 ("POSIX paths everywhere a path enters a record/table row").

**Breaking input (Windows-only):** any checkout on Windows with orphan docs in `docs/` — the Orphans section in the committed `docs/INDEX.md` artifact would carry `docs\issues\ISSUE-0001.md`-style paths.

**Why not caught by tests:** TEST-003 verifies the real-repo INDEX is POSIX-only and byte-identical. On macOS/Linux, `path.relative()` always returns POSIX paths, so the test always passes. TEST-004's CRLF fixture has no orphans (all docs have valid frontmatter), so the Orphans renderer is never exercised with a non-POSIX path.

**Pre-existing:** This gap existed before this PR. SPEC-0007's implementation plan audited `d.path` sites in `generate-docs-index.mjs`'s own scan loop but did not cover `d.rel` from the RFC-0002 audit path (`runAudit` → `scanAuditDocs`).

**Suggested fix:** In `docs-audit-core.mjs:136`, change:
```js
const rel = path.relative(root, full);
```
to:
```js
const rel = path.relative(root, full).split(path.sep).join('/');
```
Or, alternately, normalize at the renderer site in `generate-docs-index.mjs:361`.

### INFO-1 — findOpenDecisionMarker receives unnormalized content (residual CRLF exposure, out of SPEC-0007 scope)

**Category:** Style / best practice  
**File:** `docs-audit-core.mjs:192`, called from `docs-audit-core.mjs:306`

`findOpenDecisionMarker(content)` splits on `'\n'` (line 192) and receives raw (unnormalized) `content` directly from `readFileSync`. On a CRLF file, each `lines[i]` would carry a trailing `\r`. The function's regex checks (`OPEN_DECISION_COMMENT_RE`, `OPEN_DECISION_TOKEN_RE`) test mid-line patterns and are unlikely to fail on a trailing `\r`, but the behavior is untested and the function is not listed in SPEC-0007's normalization scope.

This is a pre-existing residual gap not introduced by this PR. It is **not an AC violation** for SPEC-0007 (the spec names parseFrontmatter/parseAcTable/parseReviewBy/extractReferences only).

**Suggested future fix (non-blocking):** Pass `normalizeNewlines(content)` to `findOpenDecisionMarker` in `docs-audit-core.mjs:306`.

### INFO-2 — TEST-006 negative control does not separately exercise each guard condition

**Category:** Style / test quality  
**File:** `tests/skills/test-aai-docs-audit.sh` (TEST-006 negative control)

The negative control removes two of the three legacy docs, leaving 1 legacy of 2 total. This makes `legacyCount = 1` (fails `>1`) AND `ratio = 0.5` (fails `>0.5`).

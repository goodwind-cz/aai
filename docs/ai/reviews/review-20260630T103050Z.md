### WARNING Findings

#### W1 — `findOpenDecisionMarker` does not strip inline code spans; self-triggers on SPEC-0006 line 93

**File:** `.aai/scripts/lib/docs-audit-core.mjs` line 216
**Category:** Correctness / false-positive

**The bug:** The WARNING test `const inWarning = /\bWARNING\b/.test(raw)` is applied to the raw line including any inline code spans (single-backtick `\`...\``). A line where `WARNING` appears only inside an inline code span — a legitimate informational reference, not an actual warning callout — trips the guard if an open-decision token also appears on the same line (inside or outside backticks).

**Confirmed breaking input:** SPEC-0006's own line 93 (verified by live Node.js execution):

```
   unresolved decision — e.g. a `WARNING`/`> [!WARNING]` line combined with an
```

This line has:
- `WARNING` in an inline code span (`\`WARNING\``) → `/\bWARNING\b/.test(raw) === true`
- `unresolved decision` as prose (not in backticks, at the start of the line) → `OPEN_DECISION_TOKEN_RE.test(raw) === true`

`findOpenDecisionMarker` returns `{ marker: 'WARNING with open-decision token', line: 93 }` for this input. When SPEC-0006 is eventually transitioned to `status: done`, this line causes a false-positive entry in the "Open decisions on done docs" digest section.

**Scope of impact:** Report-only — no CI gate exit codes change, no hard-fail path is triggered. The false-positive is latent (SPEC-0006 currently has `status: implementing`; the guard only runs when `status === 'done'`). However the spec's stated design intent is "narrow enough to avoid false positives on legitimate informational notes" and this is a textbook false positive.

**Suggested fix** (one line at docs-audit-core.mjs line 216):
```javascript
// Strip inline code spans before the WARNING test so `WARNING` in backticks
// does not trigger the guard (inline code is documentation, not a callout).
const stripped = raw.replace(/`[^`\n]*`/g, '');
const inWarning = /\bWARNING\b/.test(stripped) || (calloutActive && isBlockquote);
if (inWarning && OPEN_DECISION_TOKEN_RE.test(stripped)) {
```

Note: the validator already documented this as a residual risk in `last_validation.notes` and stated "Must be resolved before SPEC-0006 closes to done." This review confirms the finding and agrees with that assessment.

### INFO Findings

#### I1 — `legacy` frontmatter status vs `d.legacy` flag: potential future confusion

**File:** `.aai/scripts/generate-docs-index.mjs` lines 108 and 232
**Category:** Style / maintainability

The index generator has two distinct "legacy" concepts:
- `d.legacy = true`: set for no-frontmatter docs (exempt from coverage invariant via `!d.legacy`).
- `d.status = 'legacy'`: set for both no-frontmatter docs AND any doc with frontmatter `status: legacy`.

A doc with frontmatter `status: legacy` (a valid `DOC_STATUS_ENUM` value) has `d.legacy = false` but `d.status = 'legacy'`. It correctly lands in `coverageGaps` because there is no `byStatus('legacy')` placement section. This is intentional and the test fixture (SPEC-9003, used in TEST-002/003) deliberately exploits this to produce a zero-section doc.

The behavior is correct. However the variable names are easy to confuse: `d.legacy` (the boolean field) is NOT the same as `d.status === 'legacy'`. A comment at line 232 clarifying this distinction would reduce future confusion.

#### I2 — Fence-length check absent in `findOpenDecisionMarker`

**File:** `.aai/scripts/lib/docs-audit-core.mjs` line 202
**Category:** Minor correctness edge case

CommonMark specifies that a closing code fence must be at least as long as the opening fence (e.g., `` ``` `` can only be closed by `` ``` `` or longer, not `` `` ``). The current implementation checks only the fence character (`raw.trim()[0] === fenceCh`), not the length. An over-short run of backticks inside a fenced block would incorrectly be treated as a closing fence.

**Impact:** Very low. In the doc corpus being audited (markdown docs, not code files), nested or mismatched-length fences are essentially non-existent. A false negative here would cause content inside a fence to be scanned, potentially missing a real open-decision marker — not a false positive. No known breaking input in the current repo.

#### I3 — Close-policy is advisory-via-prompt rather than a hard gate

**File:** `.aai/VALIDATION.prompt.md` lines 127-133, `.aai/workflow/WORKFLOW.md` lines 33-40
**Category:** Design trade-off (by spec)

The "resolve-or-promote" close-policy instructs the validation agent to issue a FAIL verdict if a done-transition is requested while open decisions remain. The mechanism (`docs-audit`'s "Open decisions on done docs" section) is report-only. A validator agent that does not read the docs-audit report or ignores its output would not enforce the rule.

This is intentional per SPEC-0006 design decision 5: "Escalation from report to gate is explicitly deferred until the signal is proven clean." Not a bug; recorded for awareness.

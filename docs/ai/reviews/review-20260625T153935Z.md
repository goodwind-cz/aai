## Stage 2 — Code Quality Findings

### WARNING

**WARNING-1 — `archiveSource` can silently overwrite + then delete on a malformed approved map (data-loss / partial-failure).**
`.aai/scripts/lib/docs-canon-core.mjs:355-362`
`archiveSource` does `writeFileSync(destAbs)` then `rmSync(srcAbs)` with no guard that `destAbs` does not already exist. The already-archived case is guarded (line 349), but two failure modes remain for a **human-authored** approved map:
1. The same source listed under two domains → first domain archives + deletes the original; the second domain's `archiveSource` then throws at `readFileSync(srcAbs)` (file gone), aborting `runPhase2` mid-loop and leaving a partially-mutated tree (some sources moved, some not, canonical possibly written). No atomicity / rollback.
2. A destination collision (two distinct sources mapping to the same archive path, or a stale archive file) silently overwrites the destination, losing its prior content.
The Phase-1 *proposer* routes ambiguous sources to `unclear`, but `isApprovedMap` does not re-validate that each source appears in at most one domain, and the move stage does not refuse to overwrite an existing destination.
**Recommendation:** Before any move, validate the approved map for cross-domain source duplication and refuse on collision; have `archiveSource` throw if `destAbs` already exists (unless content-identical) rather than overwrite; consider staging all moves and only `rmSync` originals after all writes succeed (or wrap the per-domain loop so a throw does not leave a half-archived domain).

**WARNING-2 — Drift recorded-hash baseline is only the FIRST synthesis; a deliberate re-synthesis never refreshes `sourceHashes`, so a domain can stay permanently DRIFTed.**
`.aai/scripts/lib/docs-canon-core.mjs:377-393, 429`
On re-run, when `recorded && canonical exists` and sources changed, the code pushes the domain to `result.drifted` and `continue`s WITHOUT updating `d.sourceHashes` or rewriting the canonical. There is no code path in `runPhase2` that re-synthesizes a drifted domain and refreshes its baseline (the skill prompt says "re-synthesize deliberately" but the CLI offers no `--force`/`--resynth` flag to do so). Operator must hand-edit `docs-canon.map.json` to clear `sourceHashes`. This is correct for "never silently overwrite" (AC-07 satisfied) but is an ergonomic dead-end and a likely source of confusion.
**Recommendation:** Add an explicit re-synthesis path (e.g. `--resynth <domain>` or clearing recorded hashes) so drift is resolvable through the CLI, not by editing JSON.

**WARNING-3 — TEST-203 (the AC-11 e2e gate) is an assertion-free marker.**
`tests/skills/test-aai-docs-canon.sh:663-668`
`test_e2e_suite_marker` only calls `log_pass` — it asserts nothing. Because `set -e` + the per-test `log_fail`→`exit 1` model aborts the run on any earlier failure, reaching this line does imply the prior 24 tests passed, so it is not strictly tautological. But as written it adds no independent coverage and would "pass" even if its own intent regressed.
**Recommendation:** Either drop it or have it assert a concrete end-state invariant (e.g. canonical exists, archive populated, audit CLEAN) to make the e2e claim self-evidencing.

## Notes on positives

- Determinism is handled carefully throughout (sorted `readdirSync`, sorted nodes/refs/umbrellas, stable `serializeGraph` with a sorted-key replacer). Map iteration over `map.domains` preserves insertion order, which is acceptable since outputs are keyed by domain, not order-dependent.
- Body-only hashing for drift is the right call: it prevents Phase-2's own `status:`/`canonical:` frontmatter injection from being misread as a content change (verified by TEST-116/117).
- Frontmatter list parsing change is backward-compatible: only bracketed `[a, b]` now becomes an array; quoted comma-strings, `[]`, `null`, and nested blocks are unchanged, and no existing doc uses non-empty inline-bracket values.
- SEAM-3 `_archive` exclusion is depth-0 correct and prunes the whole subtree before recursion; `docs/canonical` remains scanned so canonical provenance is validated.
- Phase 1 / Phase 2 gate is enforced in both the pure function (`isApprovedMap`) and the CLI, and tested at both levels.
